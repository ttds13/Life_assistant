# 生活助手小程序 MVP 阶段实施计划

## 1. 文档目的

本文档用于指导生活助手小程序第一版 MVP 的实际开发工作。

本计划不直接进入编码，而是先把第一版拆成可逐个完成、可逐个验收、可持续扩展的阶段。每个阶段都包含业务目标、具体任务、前端工作、后台工作、后端工作、数据设计、测试重点、验收标准和后续预留点。

第一版的核心目标是完成一个可商用的订单闭环：

```txt
用户登录
-> 浏览服务
-> 预约下单
-> 微信支付
-> 后台派单
-> 师傅接单
-> 上门服务
-> 用户确认和评价
-> 售后处理
-> 平台查看数据
```

## 2. 第一版建设原则

## 2.1 优先完成业务闭环

第一版不追求功能大而全，优先保证以下闭环真实可用：

- 用户可以下单。
- 用户可以付款。
- 管理员可以派单。
- 师傅可以接单和完成服务。
- 用户可以查看订单状态。
- 用户可以评价和投诉。
- 管理员可以处理订单和售后。
- 平台可以看到基础经营数据。

## 2.2 第一版先人工，后续再自动化

以下能力第一版先做人工版本：

- 派单：先后台人工派单，后续再做自动派单。
- 结算：先后台记录收入和提现申请，后续再接自动打款。
- 客服：先做售后工单和后台处理，后续再做客服协同和智能客服。
- 营销：先做优惠券，后续再做复杂活动引擎。

## 2.3 复杂需求只预留入口

第一版暂不完整实现，但要预留数据结构和模块边界：

- 支付宝支付
- 自动派单
- 多城市
- 多门店
- 企业客户
- 积分等级
- 自动提现
- 复杂会员体系
- 复杂价格规则
- 复杂 BI 报表

## 3. 阶段总览

```txt
阶段 0：产品和技术准备
阶段 1：项目工程化和基础架构
阶段 2：账号、角色和权限
阶段 3：服务分类、服务项目和价格
阶段 4：用户端预约下单
阶段 5：订单状态机和微信支付
阶段 6：后台订单调度和人工派单
阶段 7：服务人员端任务履约
阶段 8：评价、投诉和售后工单

```

建议每个阶段完成后都进行一次小范围验收，不要等全部开发完成后再联调。

## 4. 阶段 0：产品和技术准备

## 4.1 阶段目标

在正式编码之前，把业务边界、角色、流程、技术栈、部署方式和开发顺序确认清楚，避免边写代码边反复推翻。

## 4.2 具体任务

- 明确第一版服务类型，例如保洁、维修、家电清洗。
- 确认第一版服务城市和服务区域。
- 确认第一版订单模式：用户预约，后台人工派单，师傅接单服务。
- 确认第一版支付模式：先接微信支付。
- 确认第一版用户端和师傅端是否共用一个小程序。
- 确认后台管理员角色：超级管理员、运营、客服、财务。
- 确认服务人员角色：普通师傅、组长预留。
- 确认第一版不做自动派单、不做自动提现、不做复杂积分。
- 准备微信小程序账号。
- 准备微信支付商户号。
- 准备服务器、数据库、对象存储和域名。
- 准备用户协议、隐私政策、服务协议、退款规则初稿。

## 4.3 产出物

- 第一版功能范围清单。
- 第一版暂缓功能清单。
- 角色和权限清单。
- 核心业务流程图。
- 页面清单。
- 数据库初稿。
- 接口模块清单。

## 4.4 验收标准

- 能清楚说明第一版上线后用户如何下单、平台如何派单、师傅如何服务。
- 能清楚说明第一版哪些功能做，哪些功能不做。
- 能清楚说明三端分别有哪些页面。
- 能清楚说明每个业务状态如何流转。

## 4.5 后续预留点

- 多城市：服务和订单表预留 `city_code`。
- 多门店：用户、员工、订单预留 `store_id`。
- 多角色：后台权限使用 RBAC，不写死管理员权限。
- 多支付：支付记录表使用 `channel` 字段，不把微信支付写死到订单表。

```txt

in_service         服务中
pending_confirm    待确认
completed          已完成
cancelled          已取消
```

```txt

refunded           已退款
after_sale         售后中
```

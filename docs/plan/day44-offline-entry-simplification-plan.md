# Day44 线下客户与订单录入简化改造计划

更新时间：2026-07-09  
目标版本：`1.0.3` 后续迭代  
目标模块：Admin 端用户管理、订单管理、服务管理、会员卡、财务闭环  
核心目标：把线下录入从“手填 ID + 多字段表单”改成“少填字段 + 可视化选择 + 自动补全”，降低管理员录入成本和录错风险。

## 1. 背景问题

当前后台已经支持线下客户、线下订单、线下收款、自动派单、师傅履约、财务统计闭环，但实际使用上仍然偏技术化。

现状问题：

1. 新增客户表单字段过多。
2. 录入客户时需要管理员理解来源、城市编码、联系人、联系电话、地址标题等字段。
3. 新增订单时需要手填 `用户ID`、`服务ID`、`地址ID`、`用户卡ID`。
4. ID 手填不直观，容易填错用户、服务、地址或会员卡。
5. 服务价格、服务时长、预约结束时间、客户联系电话等本来可以自动带出，但现在仍需要人工输入或确认。
6. 线下订单录入没有形成足够明确的“一步步选择”体验，管理员需要同时理解客户、地址、服务、支付、会员卡、派单规则。

Day44 要固定的方向：

```text
录入客户：只让管理员填昵称、手机号、地址，其他字段自动补齐。
录入订单：不再让管理员手填 ID，而是可视化选择已有客户、已有服务、已有地址、已有会员卡。
```

## 2. 改造原则

1. 后台仍然使用统一的 `User`、`Address`、`Service`、`Order`、`Payment`、`UserMemberCard` 数据模型。
2. 不改变现有订单、支付、派单、履约、财务闭环，只优化录入入口。
3. 管理员不应该直接输入内部 ID，除非进入高级调试模式。
4. 客户手机号仍然是线下客户去重的第一标识。
5. 客户、服务、地址、会员卡必须从可视化列表中选择，选择后再由系统写入对应 ID。
6. 表单默认值由系统生成，管理员只确认业务含义。
7. 订单金额默认来自服务价格，只有有权限的管理员才能手动覆盖金额，并必须写备注。
8. 会员卡抵扣只能选择当前客户已有的可用会员卡，不能手填其他客户的卡 ID。

## 3. 目标体验

### 3.1 简化后的线下客户录入

入口：用户管理 / 用户列表 / 新增客户

管理员只需要看到以下主字段：

| 字段 | 是否必填 | 规则 |
| --- | --- | --- |
| 手机号 | 必填 | 作为客户唯一识别，输入后自动检查是否已存在 |
| 客户昵称/姓名 | 可选 | 不填时默认使用手机号；也作为联系人默认值 |
| 服务地址 | 可选但建议填 | 一行或多行输入，系统自动生成默认服务地址 |
| 门牌号 | 可选 | 单独记录，方便上门 |
| 备注 | 可选 | 线下来源、销售、特殊说明 |

自动填写规则：

| 字段 | 自动值 |
| --- | --- |
| `User.source` | 默认 `offline` |
| `User.status` | 默认启用 |
| `User.nickname` | 昵称为空时使用手机号 |
| `User.cityCode` | 优先从默认城市配置读取；没有配置则为空 |
| `Address.ownerType` | `user` |
| `Address.addressType` | `service` |
| `Address.contactName` | 优先客户昵称，否则手机号 |
| `Address.contactPhone` | 客户手机号 |
| `Address.addressTitle` | 从地址前 20 个字符或小区名自动生成 |
| `Address.detailAddress` | 管理员填写的服务地址 |
| `Address.isDefault` | `true` |
| `Address.source` | `manual` |
| `Address.coordinateType` | 默认 `gcj02` |
| `Address.mapProvider` | 默认 `manual` |

手机号已存在时：

1. 不创建重复客户。
2. 页面显示“已存在客户”提示。
3. 展示客户姓名、手机号、已有地址数量、最近订单。
4. 如果本次填写了新地址，则只补充地址。
5. 点击确认后进入客户详情或继续创建订单。

### 3.2 简化后的线下订单录入

入口：订单管理 / 服务预约订单 / 录入外来订单

新订单弹窗改成四步或四块区域：

```text
选择客户 -> 选择服务 -> 选择地址和预约时间 -> 选择收款方式并创建订单
```

#### 第一步：选择客户

交互要求：

1. 搜索框支持输入手机号、姓名。
2. 远程搜索已有客户。
3. 搜索结果显示：客户姓名、手机号、来源、默认地址、历史订单数。
4. 点击客户卡片即选中客户。
5. 如果搜索不到，提供“快速新增客户”按钮。
6. 快速新增只填手机号、昵称、地址，不暴露 `用户ID`。

选中客户后自动带出：

1. `userId`。
2. 客户手机号。
3. 默认联系人。
4. 默认服务地址。
5. 客户已有会员卡。
6. 客户已有优惠券或可用权益后续可扩展展示。

#### 第二步：选择服务

交互要求：

1. 不再手填 `服务ID`。
2. 使用服务选择器：支持按分类、服务名称、状态搜索。
3. 列表展示服务名称、分类、价格、时长、服务单位、是否支持会员卡。
4. 仅展示上架且可下单的服务。
5. 点击服务卡片或表格行后选中服务。

选中服务后自动带出：

| 字段 | 自动值 |
| --- | --- |
| `serviceId` | 所选服务 ID |
| `originalAmount` | 服务基础价格 |
| `payableAmount` | 默认等于服务基础价格 |
| 预约结束时间 | 预约开始时间 + 服务时长 |
| 服务规格 | 服务时长、单位、会员卡计费规则 |
| 会员卡可用性 | 根据客户已持卡和服务规则自动筛选 |

#### 第三步：选择地址和预约时间

地址选择要求：

1. 默认选中客户默认服务地址。
2. 展示客户已有地址卡片，不手填 `地址ID`。
3. 每张地址卡显示联系人、电话、详细地址、门牌、默认标识。
4. 可以快速新增服务地址。
5. 新增地址只填：联系人、电话、详细地址、门牌。
6. 联系人默认等于客户昵称，联系电话默认等于客户手机号。

预约时间要求：

1. 选择预约开始时间。
2. 结束时间默认按服务时长自动计算。
3. 管理员可以手动调整结束时间。
4. 如果结束时间早于开始时间，前端阻止提交。

#### 第四步：选择收款方式

支付方式改成按钮或分段控件：

| 方式 | 含义 | 创建后状态 |
| --- | --- | --- |
| 线下已收款 | 已经现金、转账、门店收款 | `pending_dispatch` |
| 待线下收款 | 先预约，后续再确认收款 | `pending_payment` |
| 会员卡抵扣 | 使用客户已有会员卡权益 | `pending_dispatch` |

线下已收款：

1. 金额默认等于服务价格。
2. 收款备注默认生成，例如“线下录入已收款”。
3. 如管理员修改金额，必须填写修改原因。

待线下收款：

1. 创建后进入待支付。
2. 订单详情保留“确认线下收款”按钮。
3. 未确认收款前不能派单和自动派单。

会员卡抵扣：

1. 只能从当前客户可用会员卡中选择。
2. 只显示适用于当前服务的会员卡。
3. 卡片展示卡名、剩余额度、冻结额度、有效期、预计扣减。
4. 不允许手填 `UserMemberCard ID`。

## 5. API 与数据支持计划

优先复用现有接口，不够用时补充轻量接口。

### 5.1 可复用接口

现有可用能力：

1. Admin 创建用户：`POST /api/admin/users`
2. Admin 创建订单：`POST /api/admin/orders`
3. Admin 地址管理：`/api/admin/users/:id/addresses`
4. Admin 订单列表、详情、账务检查。
5. Admin 服务资源列表。
6. Admin 用户会员卡列表。

### 5.2 建议新增或封装的轻量接口

如果现有资源接口查询体验不够稳定，建议补充以下接口：

| 接口 | 用途 |
| --- | --- |
| `GET /api/admin/users/options?keyword=` | 客户远程搜索，返回姓名、手机号、默认地址、订单数 |
| `GET /api/admin/services/options?keyword=&categoryId=` | 服务选择器，返回价格、时长、状态、会员卡规则摘要 |
| `GET /api/admin/users/:id/order-entry-profile` | 订单录入所需客户资料：默认地址、地址列表、可用会员卡 |
| `GET /api/admin/users/:id/member-cards/options?serviceId=` | 当前服务下可用会员卡 |
| `POST /api/admin/users/quick-offline` | 快速新增线下客户，后端负责默认值 |

后端默认值应尽量放在服务层兜底，前端只负责展示和辅助填写。

## 6. 表单默认值与校验规则

### 6.1 客户校验

| 场景 | 规则 |
| --- | --- |
| 手机号为空 | 禁止提交 |
| 手机号已存在 | 不创建重复用户，提示复用 |
| 昵称为空 | 使用手机号 |
| 地址为空 | 允许只创建客户，但创建订单时必须补地址 |
| 地址存在 | 自动创建服务地址 |

### 6.2 订单校验

| 场景 | 规则 |
| --- | --- |
| 未选择客户 | 禁止提交 |
| 未选择服务 | 禁止提交 |
| 未选择地址 | 禁止提交 |
| 未选择预约时间 | 禁止提交 |
| 结束时间早于开始时间 | 禁止提交 |
| 待收款订单 | 创建后为 `pending_payment` |
| 未收款订单派单 | 必须拦截 |
| 线下已收款 | 必须生成 offline payment |
| 会员卡抵扣 | 必须选择当前客户可用会员卡 |
| 手动改金额 | 必须填写后台备注或金额修改原因 |

### 6.3 防错设计

1. 所有 ID 字段只在隐藏字段中保存。
2. 选中对象用卡片展示，不让管理员记 ID。
3. 客户切换后，清空地址、会员卡、订单金额确认状态。
4. 服务切换后，重新计算金额、时长、会员卡可用列表。
5. 支付方式切换后，清理不相关字段。
6. 提交前展示摘要确认。

## 7. 实施步骤

### 阶段一：客户快速录入

1. 调整新增客户弹窗字段。
2. 增加手机号防抖查询。
3. 已存在客户时展示复用卡片。
4. 自动填充联系人、联系电话、地址标题、来源、默认地址。
5. 增加“保存并创建订单”入口。
6. 验证重复手机号不会生成重复客户。

验收标准：

1. 只填手机号即可创建客户。
2. 填手机号、昵称、地址即可创建客户和默认地址。
3. 手机号已存在时不会创建重复用户。
4. 创建后能在用户列表和地址管理中查到。

### 阶段二：订单客户选择器

1. 在录入订单弹窗中增加客户搜索框。
2. 搜索结果展示客户卡片。
3. 选择客户后加载地址、会员卡、历史订单摘要。
4. 支持快速新增客户后自动选中。
5. 移除常规表单中的手填用户 ID。

验收标准：

1. 管理员通过手机号或姓名能选中已有客户。
2. 新客户可在订单弹窗内快速创建。
3. 创建订单 payload 中仍正确提交 `userId`。

### 阶段三：服务可视化选择器

1. 增加服务选择弹窗或下拉卡片。
2. 支持按服务名、分类搜索。
3. 展示价格、时长、服务状态。
4. 只允许选择上架服务。
5. 选择服务后自动带出价格和预约结束时间。
6. 移除常规表单中的手填服务 ID。

验收标准：

1. 管理员不需要知道服务 ID。
2. 选中服务后订单金额默认正确。
3. 预约结束时间按服务时长自动计算。

### 阶段四：地址与会员卡选择

1. 选中客户后展示地址卡片。
2. 默认选中默认服务地址。
3. 支持快速新增地址。
4. 会员卡支付时展示当前客户可用会员卡。
5. 根据服务规则过滤不可用会员卡。
6. 移除常规表单中的手填地址 ID 和用户卡 ID。

验收标准：

1. 地址只能从客户地址中选择或现场新增。
2. 会员卡只能从当前客户可用卡中选择。
3. 不会出现把 A 客户会员卡用到 B 客户订单的情况。

### 阶段五：提交摘要与闭环验证

1. 提交前展示订单摘要。
2. 创建成功后跳转订单详情或提示进入待派单。
3. 待收款订单仍进入 `pending_payment`。
4. 线下已收款订单生成支付记录并进入 `pending_dispatch`。
5. 会员卡订单生成冻结流水并进入 `pending_dispatch`。
6. 自动派单、师傅通知、履约、账务检查保持原闭环。

验收标准：

1. 线下已收款订单能直接派单。
2. 待收款订单未收款前无法派单。
3. 确认收款后能自动派单。
4. 订单详情账务检查通过。
5. 财务统计能统计线下订单和线下支付。

## 9. 风险与处理

| 风险 | 处理方式 |
| --- | --- |
| 前端只隐藏 ID，但接口仍可传错 ID | 后端校验地址和会员卡必须属于当前用户 |
| 服务价格被误改 | 默认不允许改金额；需要权限和原因 |
| 客户切换后仍保留旧地址或旧会员卡 | watcher 中强制清空相关选择 |
| 会员卡不适用服务 | 后端按服务规则二次校验 |
| 重复手机号导致客户重复 | 后端继续按手机号复用 |
| 地址信息太简略影响上门 | 地址输入保留详细地址和门牌号，并支持备注 |
| 默认城市不准确 | 城市信息放到高级选项，必要时人工调整 |

## 10. 最终验收清单

1. 客户录入页面只需要手机号、昵称、地址即可完成常规录入。
2. 客户手机号重复时不会创建重复用户。
3. 订单录入页面不再要求管理员手填用户 ID、服务 ID、地址 ID、用户卡 ID。
4. 管理员能通过搜索和卡片选择客户。
5. 管理员能通过服务列表选择服务。
6. 管理员能选择客户已有地址或快速新增地址。
7. 会员卡抵扣只能选择当前客户可用会员卡。
8. 服务价格、服务时长、预约结束时间可以自动带出。
9. 线下已收款订单能生成支付记录并进入待派单。
10. 待线下收款订单未收款前不能派单。
11. 确认线下收款后能进入待派单并支持自动派单。
12. 派单后师傅通知正常生成。
13. 师傅履约、用户确认、师傅收入、积分流水、财务统计闭环不被破坏。
14. 测试数据可保留查看，也可一键清理。

## 11. Day44 结论

Day44 的核心不是新增复杂业务，而是把已有闭环从“技术人员可操作”调整为“普通管理员可操作”。

必须改掉的关键点：

1. 客户录入不再暴露过多字段。
2. 订单录入不再让管理员手填内部 ID。
3. 客户、服务、地址、会员卡全部可视化选择。
4. 价格、时长、来源、联系人、默认地址等由系统自动补齐。
5. 后端继续做归属校验和闭环校验，保证方便的同时不牺牲数据准确性。

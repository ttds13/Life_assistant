# Day47 Admin 会员卡时长可视化修改计划

更新时间：2026-07-13  
目标模块：Admin 端会员卡购买订单、用户会员卡、会员卡流水、订单详情  
核心目标：把后台“编辑会员卡购买订单”的操作，调整为“可视化修改指定用户持有的指定会员卡时长和核销结果”，避免管理员误以为改购买订单就能改变用户卡余额。

## 1. 当前问题

当前 Admin 端会员卡购买订单中的“编辑”已经不匹配真实业务需求。

会员卡购买订单本质上是交易记录，表示用户在哪个时间、以什么金额、通过什么渠道购买了哪张会员卡模板。订单完成后，系统会生成一张用户实际持有的会员卡，也就是 `UserMemberCard`。

因此，后台后续要调整的对象不应该是购买订单，而应该是用户已经持有的那张会员卡。

必须明确拆分：

| 对象 | 业务含义 | 是否允许直接改时长 |
| --- | --- | --- |
| 会员卡模板 | 后台配置的商品，例如 120 分钟护理卡 | 不允许影响已售卡 |
| 会员卡购买订单 | 购买、收款、发卡记录 | 不建议编辑余额 |
| 用户会员卡 | 某个用户实际持有的一张卡 | 允许通过后台调整 |
| 会员卡流水 | 发卡、冻结、核销、释放、后台调整记录 | 只追加，不直接修改 |

## 2. 目标闭环

后台需要形成一个清晰的可视化闭环：

```text
管理员进入会员卡购买订单
-> 查看该订单发放出的用户会员卡
-> 点击“管理用户会员卡”
-> 打开该用户这张卡的余额详情
-> 点击“调整时长 / 手动核销”
-> 填写调整方式、分钟数、原因
-> 系统写入会员卡流水和审计日志
-> 用户端、师傅端、Admin 端看到最新余额
```

最终后台要能完成三类操作：

1. 增加时长：客户补偿、赠送、售后修正。
2. 减少时长：线下已使用、人工核销、异常扣除。
3. 改为指定剩余时长：历史数据修正、客服确认后的余额校准。

## 3. 页面入口调整

### 3.1 会员卡购买订单列表

位置：

```text
Admin -> 订单管理 -> 会员卡购买订单
```

调整要求：

1. 保留订单查看能力。
2. 弱化或移除不适合的“编辑订单”入口。
3. 新增明显操作按钮：
   - 查看购买订单
   - 查看发放会员卡
   - 管理用户会员卡

列表中建议展示：

| 字段 | 说明 |
| --- | --- |
| 订单号 | 购买订单编号 |
| 用户 | 用户昵称 / 手机号 |
| 购买会员卡 | 模板名称 |
| 支付金额 | 实付金额 |
| 支付渠道 | 微信 / 线下 |
| 发卡状态 | 已发卡 / 未发卡 / 发卡失败 |
| 用户会员卡 | 已生成的用户会员卡 ID 或卡名称 |
| 当前剩余 | 该用户会员卡当前剩余时长 |
| 已冻结 | 被预约占用的时长 |
| 可用余额 | 当前可继续预约的时长 |

### 3.2 会员卡购买订单详情

订单详情中新增“发放会员卡”区块：

```text
发放会员卡
- 用户会员卡 ID
- 会员卡名称
- 卡类型：时长卡 / 次卡
- 总剩余：xx 分钟
- 已冻结：xx 分钟
- 可用余额：xx 分钟
- 状态：正常 / 已用完 / 停用 / 已退款
- 操作：管理用户会员卡
```

这里不直接放一个容易误解的“编辑订单”，而是明确跳转到“用户会员卡详情”。

### 3.3 用户会员卡列表

位置：

```text
Admin -> 营销管理 -> 用户会员卡
```

这是后续调整时长的主入口。

列表行操作增加：

1. 查看详情。
2. 调整时长。
3. 手动核销。
4. 查看流水。
5. 查看来源订单。

## 4. 可视化弹窗设计

### 4.1 统一弹窗：调整会员卡时长

弹窗标题：

```text
调整用户会员卡时长
```

顶部展示当前卡信息：

```text
用户：张三 / 151xxxx2252
会员卡：120分钟护理卡
来源订单：MC202607130001
当前总剩余：90 分钟
已冻结：30 分钟
可用余额：60 分钟
状态：正常
```

操作方式使用单选切换：

| 操作方式 | 说明 |
| --- | --- |
| 增加时长 | 在当前剩余基础上增加分钟数 |
| 扣除时长 | 在当前剩余基础上减少分钟数 |
| 改为指定剩余 | 直接把总剩余改为指定分钟数 |

输入项：

| 字段 | 规则 |
| --- | --- |
| 调整分钟数 | 增加 / 扣除时填写，必须大于 0 |
| 目标剩余分钟数 | 改为指定剩余时填写，不能小于已冻结 |
| 调整原因 | 必填，例如客户补偿、线下已使用、余额修正 |
| 内部备注 | 可选，给后台人员追踪 |

底部实时预览：

```text
调整前：总剩余 90 分钟，已冻结 30 分钟，可用 60 分钟
本次变化：-20 分钟
调整后：总剩余 70 分钟，已冻结 30 分钟，可用 40 分钟
```

提交按钮文案：

```text
确认调整并记录流水
```

### 4.2 手动核销弹窗

手动核销是“扣除时长”的业务化入口，适合管理员理解。

弹窗标题：

```text
手动核销会员卡
```

场景说明：

```text
用于线下已经服务但没有系统订单、异常补录消费、客服确认后人工扣除余额。
```

输入项：

| 字段 | 规则 |
| --- | --- |
| 核销分钟数 | 必填，必须大于 0 |
| 核销原因 | 必填 |
| 关联说明 | 可选，例如线下服务时间、客服工单号 |

系统处理：

1. 本质仍调用后台调整接口。
2. `units` 记录为负数。
3. 流水类型可以继续使用 `admin_adjust`，备注中明确“手动核销”。
4. 后续如要更细，可以新增 `admin_consume` 流水类型。

## 5. 后端接口计划

当前已有基础接口：

```text
POST /api/admin/user-member-cards/:id/adjust-time
```

继续沿用该接口，但前端需要把操作语义包装清楚。

请求示例：

```json
{
  "mode": "delta",
  "deltaUnits": -20,
  "reason": "手动核销：用户线下已使用20分钟",
  "remark": "客服确认，线下服务已完成"
}
```

指定剩余示例：

```json
{
  "mode": "target",
  "targetRemainingUnits": 90,
  "reason": "余额修正：按客户购买记录校准为90分钟"
}
```

后端必须保证：

1. 只能修改 `UserMemberCard`，不能直接修改会员卡模板。
2. 调整后 `remainingUnits >= frozenUnits`。
3. 调整后同步更新 `remainingTimes`。
4. 调整后写入 `MemberCardRecord`。
5. 调整后写入 Admin 审计日志。
6. 返回调整后的完整卡详情。

## 6. 数据展示规则

后台统一展示四个关键值：

| 展示字段 | 数据来源 | 说明 |
| --- | --- | --- |
| 总剩余 | `remainingUnits` | 卡内当前总余额，包含冻结部分 |
| 已冻结 | `frozenUnits` | 已被预约订单占用但还未最终核销 |
| 可用余额 | `remainingUnits - frozenUnits` | 当前还能继续预约使用的余额 |
| 已核销 | 流水中 `consume` 和人工扣除汇总 | 展示历史消耗 |

注意：

1. 后台不要只展示“可用余额”，否则管理员会误以为卡内真实剩余减少了。
2. 有冻结时，禁止把总剩余改到小于冻结值。
3. 人工扣除不是退款，不改变购买订单金额。
4. 退款逻辑仍走退款模块，不通过“调整时长”替代。

## 7. 流水与审计

每一次后台调整必须产生记录。

会员卡流水显示：

| 字段 | 示例 |
| --- | --- |
| 流水类型 | 后台调整 |
| 变动额度 | -20 分钟 |
| 变动前 | 90 分钟 |
| 变动后 | 70 分钟 |
| 操作方 | admin |
| 操作人 | 管理员 ID / 名称 |
| 原因 | 手动核销：用户线下已使用20分钟 |
| 时间 | 2026-07-13 12:00 |

审计日志记录：

```text
action: user-member-card:time:adjust
targetType: user_member_card
targetId: 用户会员卡 ID
beforeUnits: 调整前
afterUnits: 调整后
frozenUnits: 当前冻结
reason: 调整原因
adminId: 操作管理员
```

## 8. 前端实施步骤

### 步骤 1：调整会员卡购买订单页面操作

1. 在会员卡购买订单列表增加“管理用户会员卡”按钮。
2. 如果订单未发卡，按钮置灰并提示“订单完成发卡后才能管理余额”。
3. 点击后跳转到用户会员卡详情，或直接打开用户会员卡调整弹窗。
4. 删除或弱化不适合的“编辑购买订单”操作。

验收标准：管理员不会再通过编辑购买订单去改卡余额。

### 步骤 2：补全订单详情中的发卡信息

1. 订单详情展示 `grantedUserMemberCardId`。
2. 展示该卡当前余额、冻结、可用。
3. 提供“管理用户会员卡”和“查看会员卡流水”入口。

验收标准：从某个会员卡购买订单能直接定位到用户实际持有的那张卡。

### 步骤 3：优化用户会员卡调整弹窗

1. 把当前“增加/扣减”拆成更直观的三种操作：增加时长、扣除时长、改为指定剩余。
2. 增加实时预览。
3. 原因必填。
4. 有冻结时展示红色提示。
5. 提交后刷新用户会员卡列表、订单详情和流水列表。

验收标准：管理员不用理解数据库字段，也能正确完成余额调整。

### 步骤 4：新增手动核销入口

1. 在用户会员卡列表和详情中增加“手动核销”按钮。
2. 弹窗只允许输入正数核销分钟。
3. 提交时转换为 `deltaUnits = -核销分钟数`。
4. 原因前缀统一为“手动核销：”。

验收标准：线下已服务但无订单的场景，可以被后台规范扣除并留痕。

### 步骤 5：会员卡流水筛选和展示增强

1. 流水类型加入“后台调整”。
2. 备注中高亮展示调整原因。
3. 从用户会员卡详情可以只看当前卡流水。
4. 从购买订单详情可以查看发卡和后续调整流水。

验收标准：每一次人工操作都能被追溯。

## 9. 后端核查步骤

1. 确认 `/api/admin/user-member-cards/:id/adjust-time` 已部署到云端。
2. 确认调整接口能返回用户、手机号、卡名、余额、冻结、可用。
3. 确认负数扣除、正数增加、指定剩余都能写入流水。
4. 确认调整后小程序卡包能看到最新余额。
5. 确认调整后订单详情中的会员卡余额同步更新。

## 10. 测试计划

### 10.1 正常增加时长

1. 找到一个用户已有的 120 分钟卡。
2. 后台增加 30 分钟，原因填写“客户补偿”。
3. 校验用户会员卡余额增加。
4. 校验流水出现 `+30`。
5. 校验用户端卡包同步显示。

### 10.2 手动核销时长

1. 找到一张可用余额大于 30 分钟的卡。
2. 后台手动核销 30 分钟。
3. 校验总剩余减少 30。
4. 校验可用余额减少 30。
5. 校验流水备注为“手动核销”。

### 10.3 改为指定剩余

1. 将某张卡指定剩余改为 90 分钟。
2. 校验 `remainingUnits = 90`。
3. 校验流水记录变动前后。
4. 校验审计日志记录管理员操作。

### 10.4 冻结保护

1. 找一张有冻结时长的卡，例如总剩余 100，已冻结 60。
2. 尝试把总剩余改为 50。
3. 后端必须拒绝。
4. 前端必须提示“总剩余不能小于已冻结时长”。

### 10.5 购买订单入口联动

1. 打开会员卡购买订单详情。
2. 点击“管理用户会员卡”。
3. 成功定位到该订单发放出的用户会员卡。
4. 调整后回到订单详情，余额同步刷新。

## 11. 交付验收标准

1. 会员卡购买订单不再承担修改余额的职责。
2. 后台能从购买订单直接进入用户会员卡管理。
3. 后台能对指定用户、指定会员卡做增加时长、扣除时长、指定剩余。
4. 后台能对指定用户、指定会员卡做手动核销。
5. 每次调整都有会员卡流水。
6. 每次调整都有 Admin 审计日志。
7. 有冻结时不能把总剩余改到小于冻结值。
8. 用户端卡包、师傅端订单详情、Admin 用户会员卡显示同一套最新余额。

## 12. 不做的事

本次不通过“编辑会员卡购买订单”修改余额。

本次不直接修改会员卡模板来影响历史已售卡。

本次不把人工扣除等同于退款。退款仍然走退款闭环。

本次不删除已有流水。余额修正只追加新流水。

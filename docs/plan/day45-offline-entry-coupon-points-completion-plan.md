# Day45 线下录入订单优惠券与积分闭环补全计划

更新时间：2026-07-09  
目标版本：`1.0.3` 后续迭代  
目标模块：Admin 端订单录入、会员卡购买订单、优惠券、积分、财务统计  
核心目标：修补线下录入订单在优惠券和积分上的缺口，让后台录入的服务预约订单、会员卡购买订单和小程序订单一样形成完整账务闭环。

## 1. 背景问题

Day44 已经把线下客户、服务预约订单、会员卡购买订单改成可视化录入，但当前仍有两个明显缺口：

1. 后台录入服务预约订单时，管理员不能选择当前用户可用优惠券。
2. 后台录入会员卡购买订单时，没有明确优惠券和积分的后续处理规则。
3. 线下已收款、待线下收款确认后，应同时处理支付、优惠券、积分、财务统计。
4. 如果只允许管理员手动改价，不锁定用户优惠券，会导致订单显示有优惠，但用户券流水和财务核对不完整。

Day45 要补齐的方向：

```text
选客户 -> 选服务/会员卡 -> 选择可用优惠券 -> 系统计算应收金额 -> 收款后核销优惠券 -> 按实付金额发放积分
```

## 2. 业务规则

### 2.1 服务预约订单

适用订单：

1. 线下已收款服务预约订单。
2. 待线下收款服务预约订单。

规则：

1. 只有现金支付类服务订单可以使用优惠券。
2. 会员卡抵扣订单不能使用优惠券，也不能再确认现金收款。
3. 优惠券只能从当前用户名下可用券中选择。
4. 选择优惠券后，应收金额由系统计算，不再靠管理员手填折扣。
5. 线下已收款订单创建成功后，立即核销优惠券并按实付金额发积分。
6. 待线下收款订单创建后先锁定优惠券，不发积分；确认线下收款后核销优惠券并发积分。

### 2.2 会员卡购买订单

适用订单：

1. 后台录入的线下会员卡购买订单。
2. 后台录入的待线下收款会员卡购买订单。

规则：

1. 买会员卡走“会员卡购买订单”入口。
2. 用会员卡预约服务走“服务预约订单”的“会员卡抵扣”入口。
3. 会员卡购买订单可以选择适用于会员卡购买的优惠券。
4. 线下已收款会员卡购买订单创建后，应完成：支付流水、优惠券核销、会员卡发放、积分发放。
5. 待线下收款会员卡购买订单创建后，应锁定优惠券；确认收款后再核销优惠券、发卡、发积分。
6. 会员卡购买发积分后，后续使用该会员卡预约服务时不再因为会员卡抵扣重复发积分。

### 2.3 积分规则

沿用当前积分规则：

```text
每实际支付 0.1 元积 1 分
```

补全规则：

1. 积分按实际支付金额计算，即优惠券抵扣后的金额。
2. 待收款订单未确认收款前不发积分。
3. 会员卡抵扣服务订单不发消费积分。
4. 会员卡购买订单按实际支付金额发放积分。
5. 如果后续发生退款，应按已发积分做退款扣回。

### 4.2 Admin 可用优惠券查询

建议新增轻量接口：

```text
GET /api/admin/users/:id/coupons/usable?serviceId=&amount=
```

用于后台录单时查询当前客户可用券。

返回字段：

| 字段 | 说明 |
| --- | --- |
| userCouponId | 用户券 ID |
| couponId | 优惠券模板 ID |
| name | 券名称 |
| type | 满减/折扣 |
| amount | 面额或折扣 |
| minAmount | 使用门槛 |
| discountAmount | 本单可抵扣金额 |
| expireAt | 到期时间 |

会员卡购买订单可以复用会员卡购买占位服务 `member_card_purchase` 的 `serviceId` 做适用范围校验。

### 4.3 创建线下服务预约订单

文件参考：

```text
server/src/orders/orders.service.ts
```

补全逻辑：

1. 解析 `couponId`。
2. 仅现金支付订单允许使用优惠券。
3. 使用 `CouponsService.lockCouponForOrder` 校验并锁定当前用户优惠券。
4. 由优惠券计算 `discountAmount` 和 `payableAmount`。
5. 线下已收款订单创建支付流水后，调用 `markCouponUsedForOrder`。
6. 线下已收款订单按实付金额调用 `ensureEarnedPointsForPaidOrder`。
7. 待线下收款订单只锁券，不发积分。
8. 后续确认线下收款时，再核销优惠券并发积分。

### 4.4 确认线下收款

文件参考：

```text
server/src/orders/orders.service.ts
```

补全逻辑：

1. 创建 offline 支付流水。
2. 如果订单有 `couponId`，核销对应锁定券。
3. 按实付金额发放积分。
4. 服务预约订单进入待派单。
5. 会员卡购买订单完成发卡。
6. 会员卡购买订单也按实付金额发放积分。

### 4.5 创建会员卡购买订单

文件参考：

```text
server/src/member-cards/member-cards.service.ts
```

补全逻辑：

1. 支持 `couponId`。
2. 创建订单后锁定优惠券。
3. 使用会员卡售价计算优惠金额和应收金额。
4. 线下已收款时：
   - 创建 offline 支付流水。
   - 核销优惠券。
   - 发放用户会员卡。
   - 按实付金额发积分。
   - 订单完成。
5. 待线下收款时：
   - 订单为待支付。
   - 优惠券保持锁定。
   - 不发卡、不发积分。
   - 确认收款后统一完成后续动作。

### 4.6 积分服务

文件参考：

```text
server/src/users/users.service.ts
```

当前逻辑会排除 `member_card_purchase`。Day45 需要调整为：

2. 会员卡购买现金订单也发积分。
3. 会员卡抵扣服务订单不发积分。
4. 积分备注区分：
   - 服务消费积分
   - 会员卡购买积分
5. 保持幂等：同一订单不能重复发积分。

### 4.7 财务检查

文件参考：

```text
server/src/orders/orders.service.ts
```

需要调整 `getAdminOrderAccounting`：

1. 订单有优惠券时，检查用户券状态是否正确。
2. 待支付订单优惠券应为 `locked`。
3. 已支付/已完成订单优惠券应为 `used`。
4. 已支付服务预约订单应有积分流水。
5. 已支付会员卡购买订单应有积分流水。
6. 会员卡抵扣服务订单不要求积分流水。

## 5. 退款与取消补全

涉及优惠券和积分后，必须补齐回滚规则。

| 场景 | 优惠券处理 | 积分处理 |
| --- | --- | --- |
| 待支付订单取消 | 释放锁定优惠券 | 不处理积分 |
| 已支付服务订单退款 | 优惠券按规则释放或保持已用 | 扣回已发积分 |
| 已支付会员卡购买退款 | 优惠券按规则释放或保持已用 | 扣回已发积分 |
| 会员卡抵扣服务取消 | 不涉及优惠券 | 不涉及积分 |

Day45 优先保证创建、确认收款、退款扣积分不漏；优惠券退款后是否恢复可用，按现有 `releaseCouponForRefund` 规则执行。

## 6. 前端交互规则

### 6.1 服务预约订单

展示顺序：

```text
选择客户 -> 选择服务 -> 选择地址和时间 -> 选择收款方式 -> 选择优惠券 -> 创建订单
```

校验规则：

## 7. 实施步骤

### 阶段一：后端基础能力

1. 新增 Admin 可用优惠券查询接口。
2. 给 Admin 服务预约订单 DTO 增加 `couponId`。
3. 给 Admin 会员卡购买订单 DTO 增加 `couponId`。
4. 调整积分服务，允许会员卡购买订单按实付金额发积分。

验收标准：

1. 后台能按用户、服务、金额查出可用券。
2. 会员卡购买订单可以被积分服务识别。
3. 同一订单重复调用发积分不会重复生成流水。

### 阶段二：服务预约订单优惠券闭环

1. Admin 创建服务预约订单时支持锁券。
2. 线下已收款创建时核销券并发积分。
3. 待线下收款创建时只锁券。
4. 确认线下收款时核销券并发积分。
5. 取消或退款时释放/扣回对应记录。

验收标准：

1. 线下已收款服务订单使用优惠券后，用户券为已使用。
2. 待线下收款服务订单创建后，用户券为锁定。
3. 待线下收款确认收款后，用户券变为已使用。
4. 积分按实付金额发放。

### 阶段三：会员卡购买订单优惠券与积分闭环

1. 会员卡购买订单录入支持优惠券。
2. 线下已收款时完成支付、核券、发卡、发积分。
3. 待线下收款时先锁券。
4. 确认收款后完成支付、核券、发卡、发积分。
5. 退款时扣回积分并处理会员卡回滚。

验收标准：

1. 买会员卡可选择符合规则的优惠券。
2. 实付金额正确进入支付流水。
3. 会员卡正常发放。
4. 积分按实付金额发放。
5. 财务统计能统计优惠、支付、积分。

### 阶段四：Admin 前端页面

1. 服务预约订单弹窗增加优惠券选择。
2. 会员卡购买弹窗增加优惠券选择。
3. 金额摘要增加：原价、优惠、应收、预计积分。
4. 订单详情展示优惠券和积分结果。
5. 财务检查按钮展示优惠券/积分检查结果。

验收标准：

1. 管理员不需要手动填写优惠金额。
2. 优惠券只能从当前客户可用券选择。
3. 切换客户、服务、会员卡后不会保留错误优惠券。
4. 页面展示的应收金额与后端落库一致。

## 9. 风险与处理

| 风险 | 处理方式 |
| --- | --- |
| 管理员手动改价和优惠券重复叠加 | Day45 默认不叠加；选择优惠券后金额由系统计算 |
| 待支付订单锁券后长期不支付 | 后续可加定时释放；Day45 先保证取消时释放 |
| 会员卡购买是否允许全额优惠 | 后端必须明确 0 元订单路径，不发积分 |
| 积分重复发放 | 积分服务按 `orderId + type` 幂等 |
| 会员卡购买发积分后又用卡消费 | 用会员卡抵扣服务不再发现金消费积分 |
| 前端金额被篡改 | 后端重新计算优惠和应收金额 |

## 10. 最终验收清单

1. 线下服务预约订单能选择当前用户可用优惠券。
2. 会员卡购买订单能选择当前用户可用优惠券。
3. 后端不信任前端优惠金额，统一重新计算。
4. 线下已收款订单创建后能核销优惠券。
5. 待线下收款订单创建后能锁定优惠券。
6. 确认线下收款后能核销优惠券。
7. 服务预约现金订单按实付金额发积分。
8. 会员卡购买现金订单按实付金额发积分。
9. 会员卡抵扣服务订单不重复发积分。
10. 退款后能扣回积分。
11. 财务检查能展示优惠券和积分闭环是否正常。
12. 测试脚本能跑通并清理模拟数据。

## 11. Day45 结论

Day45 的核心是把“线下录入订单”从只生成订单和支付，补全为完整的权益闭环：

```text
优惠券选择 -> 优惠金额计算 -> 优惠券锁定/核销 -> 实付金额支付 -> 积分发放 -> 财务统计
```

完成后，后台录入的服务预约订单和会员卡购买订单，在优惠券、积分、支付、会员卡和财务统计上，应与小程序订单保持同一套规则。

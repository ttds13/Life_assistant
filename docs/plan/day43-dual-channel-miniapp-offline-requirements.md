# Day43 双渠道业务分类与需求梳理

更新时间：2026-07-09  
来源：当前业务渠道梳理需求  
目标版本：`1.0.3` 后续迭代  
目标：把系统从“默认所有业务都由小程序用户主动发起”调整为“双渠道并行”：小程序自助渠道和线下手动录入渠道。两个渠道共享客户、服务、订单、会员卡、派单、师傅履约和财务统计底座，但入口、支付、确认、通知和售后责任边界必须分开。

## 1. 背景结论

当前系统的主链路已经比较完整，但整体设计仍明显偏向“小程序用户自助发起”：

```text
用户注册/登录 -> 小程序浏览服务或会员卡 -> 小程序下单/购买 -> 在线支付或会员卡抵扣 -> Admin 派单 -> 师傅履约 -> 用户确认/售后
```

实际业务需要拆成两个渠道：

| 渠道 | 业务入口 | 客户来源 | 支付方式 | 下单方式 | 派单方式 | 核心问题 |
| --- | --- | --- | --- | --- | --- | --- |
| 小程序自助渠道 | 微信小程序 | 用户主动注册/登录 | 微信支付/会员卡权益 | 用户自己创建订单 | Admin 派单或自动派单 | 当前主流程基本覆盖 |
| 线下手动录入渠道 | 门店、电话、微信私域、线下收款 | Admin 手动录入客户 | 线下收款/后台确认/后台发卡 | Admin 代客户创建订单 | Admin 派单 | 不能再假设用户必须先在小程序完成注册、购买、下单 |

Day43 要固定的核心原则：

1. 两个渠道都必须落到统一的客户主数据 `User`。
2. 两个渠道都必须落到统一的订单表 `Order`。
3. 两个渠道都必须走统一的派单、师傅履约、会员卡核销、财务统计和售后闭环。
4. 小程序渠道强调用户自助和在线支付。
5. 线下渠道强调 Admin 代录入、线下收款确认、操作审计和后续可绑定小程序。

## 2. 渠道定义

### 2.1 渠道 A：小程序自助渠道

小程序自助渠道是用户自己完成身份、购买和预约动作。

典型链路：

```text
微信登录/手机号绑定
-> 浏览服务/会员卡
-> 购买会员卡或直接预约服务
-> 微信支付/会员卡抵扣
-> 订单进入待派单
-> Admin 派单给师傅
-> 师傅接单、上门、开始、完成
-> 用户确认完成/评价/售后
```

核心数据标记：

| 对象 | 建议标记 |
| --- | --- |
| `User.source` | `miniapp` |
| `Order.source` | `miniapp` |
| `Payment.channel` | `wechat` |
| `UserMemberCard.source` | `purchase` |
| 操作人 | `user` / `system` |

小程序渠道的关键要求：

1. 用户可以自己注册、登录、绑定手机号。
2. 用户可以自己购买会员卡。
3. 用户可以自己预约服务。
4. 用户可以选择微信支付、优惠券、积分或会员卡权益。
5. 会员卡购买成功后自动发卡。
6. 会员卡预约时自动冻结额度。
7. 用户端能看到订单、会员卡、优惠券、积分、售后和通知。
8. Admin 只负责管理、派单、审核、售后和财务。

### 2.2 渠道 B：线下手动录入渠道

线下渠道是客户没有先走小程序，或者交易已在线下完成，由 Admin 把客户、购买、预约和收款录入系统。

典型入口：

1. 门店线下付款。
2. 电话预约。
3. 微信私域转账。
4. 老客户复购。
5. 线下销售人员成交。
6. 客服代下单。

典型链路：

```text
Admin 查询或新建客户
-> Admin 录入客户手机号、姓名、地址、来源备注
-> Admin 录入购买内容或服务预约
-> Admin 确认线下收款或后台发放会员卡
-> 订单进入待派单
-> Admin 派单给师傅
-> 师傅接单、上门、开始、完成
-> Admin 或客户确认完成
-> 财务统计、会员卡流水、售后记录闭环
```

核心数据标记：

| 对象 | 建议标记 |
| --- | --- |
| `User.source` | `offline` 或 `admin` |
| `Order.source` | `offline` / `phone` / `wechat_private` / `admin` |
| `Payment.channel` | `offline` |
| `UserMemberCard.source` | `admin` / `offline` |
| 操作人 | `admin` |

线下渠道的关键要求：

1. Admin 可以直接创建客户，不要求客户先登录小程序。
2. 客户可以没有 `openid`，但必须有手机号或明确的线下客户编号。
3. Admin 可以给客户创建服务预约单。
4. Admin 可以确认线下收款。
5. Admin 可以给客户发放线下购买的会员卡。
6. Admin 可以用客户已有会员卡创建预约单并冻结额度。
7. 线下订单仍必须进入统一派单和师傅履约流程。
8. 线下收款、后台发卡、后台代下单必须写审计日志。
9. 客户后续登录小程序时，应能通过手机号绑定到同一个客户主数据，而不是新建重复客户。

## 3. 不能混淆的三个概念

### 3.1 客户来源

客户来源回答的是：这个客户最早从哪里进入系统。

建议字段：

```text
User.source
```

建议值：

| 值 | 含义 |
| --- | --- |
| `miniapp` | 小程序注册 |
| `offline` | 线下客户 |
| `admin` | 后台录入 |
| `phone` | 电话预约 |
| `wechat_private` | 微信私域 |
| `channel` | 推广渠道 |

### 3.2 订单来源

订单来源回答的是：这笔订单是通过哪个入口创建的。

建议字段：

```text
Order.source
```

建议值：

| 值 | 含义 |
| --- | --- |
| `miniapp` | 用户小程序自助下单 |
| `admin` | 后台代客户下单 |
| `offline` | 线下成交后录入 |
| `phone` | 电话预约录入 |
| `wechat_private` | 私域微信成交录入 |

### 3.3 支付渠道

支付渠道回答的是：钱是怎么收的。

建议字段：

```text
Payment.channel
```

建议值：

| 值 | 含义 |
| --- | --- |
| `wechat` | 微信支付 |
| `offline` | 线下收款 |
| `mock` | 测试支付 |
| `member_card` | 权益抵扣，不是真实现金支付 |

这三个概念不能混在一起：

```text
客户来自线下，不代表每一笔订单都来自线下。
订单由 Admin 创建，不代表支付一定是线下。
支付是线下，不代表客户不能后续绑定小程序。
```

## 4. 业务对象归属

| 业务对象 | 小程序渠道 | 线下渠道 | 统一要求 |
| --- | --- | --- | --- |
| 客户 | 用户自己注册/登录 | Admin 创建或匹配客户 | 最终都落到 `User` |
| 地址 | 用户自己维护 | Admin 代录入 | 最终都落到统一 `Address` |
| 服务预约 | 用户创建 | Admin 代创建 | 最终都落到 `Order` |
| 会员卡购买 | 用户小程序购买并支付 | Admin 线下收款后发放或录入购买单 | 最终都生成 `UserMemberCard` |
| 会员卡使用 | 用户下单时选择 | Admin 代客户选择 | 都冻结、释放、核销并写流水 |
| 派单 | Admin/系统派单 | Admin/系统派单 | 统一 `OrderAssignment` 和通知 |
| 师傅履约 | 师傅端处理 | 师傅端处理 | 统一接单、上门、开始、完成 |
| 支付 | 微信支付 | 线下收款确认 | 统一 `Payment` |
| 售后退款 | 用户申请/Admin 审核 | Admin 录入/审核 | 统一退款和售后工单 |
| 财务统计 | 按微信支付统计 | 按线下收款统计 | 统一财务口径，按 source/channel 分组 |

## 5. 小程序渠道需求

### 5.1 注册与客户识别

必须支持：

1. 微信登录。
2. 手机号绑定。
3. 已有线下客户手机号登录时，合并或绑定到已有 `User`。
4. 避免同一个手机号产生多个客户。

验收标准：

```text
同一手机号的小程序注册用户和线下录入客户最终只能对应一个有效 User。
```

### 5.2 商品和服务购买

小程序自助购买分两类：

| 类型 | 订单类型 | 后续是否派单 |
| --- | --- | --- |
| 服务预约 | `service_booking` / `consultation` | 是 |
| 会员卡购买 | `member_card_purchase` | 否，支付成功后发卡 |

要求：

1. 会员卡购买订单不进入派单。
2. 服务预约订单进入待支付或待派单。
3. 会员卡预约订单不走现金支付，走冻结权益。
4. 优惠券、积分、会员卡互斥或叠加规则必须明确展示。

### 5.3 用户端闭环

小程序必须能看到：

1. 我的订单。
2. 我的会员卡。
3. 会员卡流水。
4. 优惠券。
5. 积分。
6. 地址。
7. 售后/退款。
8. 师傅履约状态。

## 6. 线下渠道需求

### 6.1 Admin 创建或匹配客户

Admin 录入线下客户时必须先查重：

```text
手机号 -> 已有 User -> 复用
手机号 -> 不存在 -> 新建 User(source=offline/admin)
```

客户字段要求：

| 字段 | 要求 |
| --- | --- |
| 手机号 | 强烈建议必填，用于后续小程序绑定 |
| 昵称/姓名 | 可填 |
| 来源 | 必填，默认 `offline` 或 `admin` |
| 地址 | 服务预约时必填 |
| 备注 | 记录线下来源、销售人员、收款说明 |

### 6.2 Admin 线下购买会员卡

线下会员卡购买有两种实现模式：

| 模式 | 说明 | 优先级 |
| --- | --- | --- |
| 发放模式 | Admin 确认客户已线下付款后，直接发放 `UserMemberCard`，流水为 `grant` | 短期优先 |
| 订单模式 | Admin 创建 `member_card_purchase` 订单，确认 offline payment 后自动发卡 | 中长期更规范 |

短期建议：

```text
Admin 发放会员卡时必须填写：
客户
会员卡模板
总额度
有效期
线下收款金额
收款方式
收款备注
```

中长期建议：

```text
线下购买会员卡也创建购买订单：
Order.orderType = member_card_purchase
Order.source = offline/admin
Payment.channel = offline
Payment.status = success
支付确认后生成 UserMemberCard
```

这样财务统计会更完整。

### 6.3 Admin 线下服务预约

Admin 代客户创建服务预约时必须支持：

1. 选择已有客户或新建客户。
2. 录入服务地址。
3. 选择服务。
4. 选择预约时间。
5. 选择支付方式：
   - 线下现金/转账收款。
   - 会员卡权益抵扣。
   - 待收款。
6. 创建订单后进入待派单或待收款。
7. 派单给师傅。

建议状态：

| 场景 | 初始状态 |
| --- | --- |
| 已线下收款服务单 | `pending_dispatch` |
| 未收款服务单 | `pending_payment` 或 `pending_offline_payment` |
| 会员卡权益预约 | `pending_dispatch` |
| 咨询单 | `pending_dispatch` |

### 6.4 线下会员卡预约

如果客户已经有会员卡，Admin 代客户创建预约时必须走同一套会员卡规则：

```text
选择客户
-> 查询客户可用会员卡
-> 选择适用会员卡
-> 后端判断服务适用性
-> 冻结会员卡额度
-> 写入订单规则快照
-> 创建待派单订单
```

不能让 Admin 手动改余额绕过冻结和流水。

### 6.5 线下收款确认

线下收款必须形成可审计记录：

```text
Payment.channel = offline
Payment.status = success
Payment.amount = 实收金额
Payment.callbackRaw / remark = 收款说明
Order.paidAt = 收款确认时间
Order.paidAmount = 实收金额
AuditLog = admin 确认收款
```

Admin 页面必须显示：

1. 谁确认了收款。
2. 确认时间。
3. 收款金额。
4. 收款方式。
5. 备注。
6. 是否进入财务统计。

## 7. 派单与师傅端统一要求

无论订单来自小程序还是线下，派单和师傅履约都必须统一。

统一链路：

```text
Order.pending_dispatch
-> Admin assign
-> Order.dispatched
-> Staff accept
-> Staff on_the_way
-> Staff in_service
-> Staff complete
-> User/Admin confirm
-> Order.completed
```

渠道差异：

| 项目 | 小程序渠道 | 线下渠道 |
| --- | --- | --- |
| 客户通知 | 小程序订单状态通知 | 可选短信/电话/Admin 手动沟通 |
| 完成确认 | 用户端确认为主 | Admin 代确认或客户后续小程序确认 |
| 售后入口 | 用户端申请 | Admin 录入售后 |
| 评价 | 用户端评价 | 可选后台代录入或不评价 |

必须统一：

1. 师傅端只看服务履约订单，不看会员卡购买订单。
2. 师傅端不关心订单是小程序还是线下，只关心服务内容、地址、时间、客户电话。
3. 派单通知必须发给师傅。
4. 履约照片、打卡、完成记录统一入库。
5. 师傅收入计算统一，但财务报表要能按订单来源区分。

## 8. Admin 端需求归类

### 8.1 客户管理

需要支持：

1. 新建线下客户。
2. 按手机号查重。
3. 客户来源筛选。
4. 客户地址管理。
5. 客户小程序绑定状态。
6. 客户历史订单、会员卡、优惠券、积分、售后统一查看。

### 8.2 订单管理

需要支持：

1. 小程序订单列表。
2. 线下订单列表。
3. 按 `Order.source` 筛选。
4. Admin 代创建订单。
5. Admin 确认线下收款。
6. Admin 派单。
7. Admin 代取消、代改期、代确认完成。
8. 订单详情展示渠道信息、收款信息和操作日志。

### 8.3 会员卡管理

需要支持：

1. 小程序购买会员卡自动发卡。
2. 线下购买会员卡后台发卡。
3. 后台发卡必须记录收款或发放原因。
4. 用户会员卡来源可筛选：`purchase`、`admin`、`offline`。
5. 会员卡流水必须区分操作人：`user`、`admin`、`system`。
6. 会员卡预约不论来自哪个渠道，都必须走冻结、释放、核销流水。

### 8.4 财务管理

财务必须按两个维度统计：

| 维度 | 字段 |
| --- | --- |
| 订单来源 | `Order.source` |
| 支付渠道 | `Payment.channel` |

报表必须能回答：

1. 小程序订单收入多少。
2. 线下订单收入多少。
3. 微信支付收入多少。
4. 线下收款收入多少。
5. 会员卡购买收入多少。
6. 会员卡权益消耗多少。
7. 退款来自哪个渠道。
8. 师傅收入对应哪些渠道订单。

## 10. 问题固定

| 编号 | 问题 | 当前表现 | 目标 |
| --- | --- | --- | --- |
| D43-CH-01 | 入口假设单一 | 系统主链路默认用户从小程序发起 | 明确小程序和线下两个入口 |
| D43-CH-02 | 客户来源和订单来源容易混淆 | `source` 字段语义不够统一 | 固定 `User.source`、`Order.source`、`Payment.channel` 三套口径 |
| D43-CH-03 | 线下购买会员卡财务闭环不够明确 | 后台发卡可以表示线下购买，但不一定有收款记录 | 建议短期发卡加收款备注，中长期走购买订单 + offline payment |
| D43-CH-04 | Admin 代客户下单边界不清 | 外来订单和后台订单需要统一归类 | 固定为线下手动录入渠道 |
| D43-CH-05 | 线下客户后续小程序绑定不明确 | 可能产生重复客户 | 手机号查重和绑定合并必须成为闭环 |
| D43-CH-06 | 财务统计缺少双维度口径 | 只看金额容易混淆渠道 | 按订单来源和支付渠道同时统计 |
| D43-CH-07 | 师傅端不应感知购买渠道 | 订单详情可能缺少来源说明 | 师傅端统一履约，Admin 端看渠道 |

## 16. Day43 实施记录

### 16.1 已实现的渠道分割闭环

1. Admin 客户录入改为按手机号复用：手机号已存在时不再 409 阻断，而是复用已有 `User`，可补建服务地址并写入 `user:reuse` 审计日志。
2. Admin 服务预约新增支付模式：
   - `offline_paid`：线下已收款，订单直接进入 `pending_dispatch`，同步生成 `Payment.channel=offline`、`PaymentNotifyLog`、积分流水和审计日志。

   - `member_card`：会员卡抵扣，创建订单时写入 `memberCardId`，冻结会员卡额度，生成 `freeze` 流水，订单进入 `pending_dispatch`。
3. Admin 代客会员卡预约的冻结流水改为 `operatorType=admin`，避免后台代操作被记录成用户自助操作。
4. Admin 发放会员卡新增来源和线下收款备注：支持 `source=admin/offline/adjust`，并把 `offlinePaymentAmount`、`paymentChannel`、`paymentRemark` 写入会员卡流水备注和审计详情。
5. Admin 订单录入页补齐来源和支付方式字段：支持线下已收款、待线下收款、会员卡抵扣、微信私域、推广渠道。
6. Admin 客户页补齐线下客户来源枚举，提示语改为“手机号存在时复用客户”。
7. Admin 财务统计页补齐 `wechat_private`、`channel` 来源筛选。

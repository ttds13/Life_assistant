# 生活助手小程序开发计划

## 1. 项目目标

本项目目标是开发一个可商用的生活服务类小程序系统，覆盖用户端、服务人员端和管理后台三类使用场景。

系统第一阶段重点满足保洁、维修、家电清洗等本地生活服务的在线预约、订单管理、人工派单、在线支付、服务过程记录、评价和售后处理能力。

作为独立开发者，第一版应优先实现可上线、可运营、可收款、可派单的 MVP，避免一开始投入过多时间在自动派单、复杂会员、复杂营销和多支付渠道上。

## 2. 技术选型

### 2.1 总体架构

```txt
life-assistant/
  miniapp/        # uni-app 小程序，第一版同时承载用户端和服务人员端
  admin-web/      # Vue3 管理后台
  server/         # NestJS 后端服务
  docs/           # 项目文档
```

第一版建议用户端和服务人员端共用一个 uni-app 小程序，通过账号角色区分入口和权限。后期如果服务人员端使用频率高、交互复杂，再拆分为独立小程序。

### 2.2 小程序端

```txt
uni-app
Vue3
TypeScript
Pinia
Vite
Wot UI 或 uView Plus
Sass
dayjs
zod 或 valibot
```

选择 uni-app 的原因：

- 和现有 JavaScript、Node.js 技术背景接近，迁移成本较低。
- 一套代码可构建微信小程序、支付宝小程序、H5 等端。
- 比微信原生小程序更适合组件化、工程化和长期维护。
- 生态成熟，适合独立开发者快速交付。

### 2.3 管理后台

```txt
Vue3
Vite
TypeScript
Element Plus
Pinia
Vue Router
Axios
ECharts
xlsx
```

后台可以基于 Vben Admin 或同类 Vue3 管理模板二次开发，减少权限、菜单、布局、表格、表单等基础工作量。

### 2.4 后端服务

```txt
NestJS
Prisma
MySQL 8 或 PostgreSQL
Redis
JWT
BullMQ
Swagger / OpenAPI
```

后端选择 NestJS 的原因：

- 基于 Node.js，符合现有技术栈。
- 模块化结构清晰，适合订单、支付、员工、售后等复杂业务。
- TypeScript 类型约束对商用系统更安全。
- 与 Prisma、Redis、队列、Swagger 集成方便。

### 2.5 基础设施

```txt
服务器：腾讯云轻量服务器 / CVM
数据库：腾讯云 MySQL / PostgreSQL
缓存：Redis
文件存储：腾讯云 COS
地图导航：腾讯位置服务或高德地图
消息通知：微信订阅消息 + 后台站内信 + 短信兜底
支付：第一版先接微信支付 V3，后续补支付宝支付
```

## 8. 订单状态机

### 8.1 主流程

```txt
待支付 -> 待派单 -> 已派单 -> 已接单 -> 上门中 -> 服务中 -> 待确认 -> 已完成
```

### 8.2 异常状态

```txt
已取消
退款中
已退款
售后中
售后完成
```

### 8.3 状态设计原则

- 所有状态变更必须写入订单状态日志。
- 不能只修改订单主表状态。
- 支付回调、取消订单、退款回调必须做幂等处理。
- 订单金额必须保存价格快照，不能依赖服务当前价格。
- 管理员手动修改状态必须记录操作人和原因。

## 10. 商用注意事项

- 微信支付必须在服务端签名和验签。
- 支付成功不能相信前端结果，必须以微信支付回调为准。
- 支付回调必须做幂等处理。
- 订单金额、优惠券、服务价格必须保存快照。
- 用户评价图片、服务过程照片建议接入内容安全审核。
- 师傅端定位、打卡和照片上传要保存时间、经纬度和操作人。
- 后台改价、派单、退款、禁用员工等操作必须写入操作日志。
- 退款规则、服务协议、隐私政策和用户协议上线前必须准备。
- 第一版提现建议使用后台线下结算记录，不建议直接接自动代付。
- 生产环境必须启用 HTTPS。
- 数据库需要定期备份。
- 管理后台账号必须使用强密码和权限控制。

## 11. 独立开发者执行建议

- 第一版不要追求功能大而全，以订单闭环和收款闭环为核心。
- 用户端和师傅端先合并在一个小程序内，降低审核和维护成本。
- 后台人工派单优先于自动派单。
- 微信支付优先于多支付渠道。
- 优惠券只做满减券或固定金额券，不做复杂活动组合。
- 会员次数卡可以先做购买、核销和记录，不做复杂成长体系。
- 数据看板先做核心指标，不做复杂 BI。
- 每完成一个业务闭环就进行真机测试，避免最后集中联调。

## 13. 第一版交付标准

达到以下标准后，可以认为第一版具备商用上线条件：

- 用户可以登录并填写地址。
- 用户可以浏览服务并预约下单。
- 用户可以完成微信支付。
- 后台可以看到订单并人工派单。
- 服务人员可以接单、打卡、上传照片并完成服务。
- 用户可以查看订单状态并评价。
- 用户可以发起售后投诉。
- 管理员可以处理投诉和评价。
- 管理员可以管理服务项目、员工和优惠券。
- 系统可以导出基础订单数据。
- 支付、订单、派单、售后等关键操作有日志。
- 小程序端在真机上完成主流程测试。

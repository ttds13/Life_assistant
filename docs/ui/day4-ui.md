# Day 4 UI 设计计划

## 1. 目标

本文件用于指导生活助手小程序 Day 4 的前端 UI 设计与页面落地。参考对象为：

- 用户端参考：https://shangmenjiazheng.xinzhiyukeji.cn/daojia_mobile/
- 师傅/服务商端参考：https://shangmenjiazheng.xinzhiyukeji.cn/laike_mobile/

本项目只参考页面结构、业务入口和交互流程，不复制参考站的代码、图片、品牌色、图标和文案。

Day 4 的 UI 目标：

```txt
用户端完成预约下单、地址、订单列表、订单详情的 UI 闭环
师傅/服务商端形成工作台、任务列表、任务详情、履约操作的 UI 方案
沉淀订单卡片、状态标签、地址卡片、底部操作栏等可复用组件
保持首页 / 订单 / 我的三 Tab 主结构清晰
避免过度营销化，优先保证服务流程清楚、价格清楚、状态清楚
```

## 2. 参考端结构提炼

### 2.1 用户端参考结构

从用户端打包路由可看出，参考产品包含以下主要模块：

```txt
首页
分类
预约
订单
我的
登录
搜索
服务详情
确认订单
立即支付
订单详情
售后
评价

优惠券
会员卡 / 次卡
钱包
附近师傅
师傅详情
设置
协议 / WebView
```

对本项目有直接参考价值的部分：

- 首页服务入口：分类宫格、推荐服务、搜索入口。
- 分类和搜索：帮助用户快速找到服务。
- 服务详情：价格、服务说明、评价、预约入口。
- 确认订单：服务信息、地址、时间、优惠、金额明细。
- 订单列表：按状态筛选，订单卡片展示下一步操作。
- 订单详情：状态突出、时间线、服务信息、支付信息、操作按钮。
- 地址管理：选择地址、新增地址、编辑地址。
- 售后和评价：Day 4 先预留入口，后续模块补齐。

暂不建议 Day 4 完整实现的部分：

- 钱包充值、提现。
- 附近师傅和师傅详情。
- 复杂会员卡购买。
- 购物车式流程。
- 大量营销专题页。

```txt

  订单详情

  评价
  佣金

```

对本项目有直接参考价值的部分：

- 工作台用数字概览突出待处理任务。
- 任务列表按状态切换：待接单、进行中、已完成。
- 任务详情突出预约时间、地址、用户电话、导航入口。
- 履约动作按状态变化：接单、拒单、上门打卡、开始服务、上传照片、完成服务。
- 服务商可预留员工管理、指派师傅、结算入口。

暂不建议 Day 4 完整实现的部分：

- 服务商独立录单流程。
- 佣金提现、银行卡绑定。
- 学习中心。
- 复杂师傅管理和分佣设置。
- 多身份认证完整审核流。

## 6. 视觉方向

### 6.1 总体风格

```txt
清晰
可信
轻运营
强流程
少装饰
状态明确
```


## 7. 用户端页面设计

### 7.1 预约下单页

路径建议：

```txt
src/pages/order/create.vue
```

页面目标：让用户确认服务、选择时间和地址，并提交订单。

页面结构：

```txt
服务摘要卡片
  服务名称
  服务封面
  起步价 / 单位
  服务说明摘要

预约信息
  预约日期
  预约时间段
  服务数量 / 面积 / 时长（按服务类型可选）

服务地址
  默认地址
  选择地址
  新增地址

备注
  用户补充说明

优惠与权益
  优惠券入口（Day 4 可先占位）
  会员次卡入口（Day 4 可先占位）

金额明细
  服务金额
  优惠金额
  应付金额

底部操作栏
  合计金额
  提交订单
```

关键交互：

- 未选择时间时，提交按钮禁用或提交时提示。
- 未选择地址时，提交按钮禁用或提交时提示。
- 金额明细只展示后端返回的计算结果，前端不自行决定最终价格。
- 提交中按钮进入 loading 状态，避免重复提交。

## 11. 订单状态 UI 映射

建议统一状态：

```txt
pending_payment    待支付      warning
pending_dispatch   待派单      warning
dispatched         已派单      primary
accepted           已接单      primary
on_the_way         上门中      primary
in_service         服务中      primary
pending_confirm    待确认      warning
completed          已完成      success
cancelled          已取消      info
refund_pending     退款中      warning
refunded           已退款      info
after_sales        售后中      error
```

UI 规则：

- 用户端展示用户能理解的状态说明。
- 师傅端展示下一步履约动作。
- 状态标签颜色必须固定，不允许每个页面单独定义。
- 订单详情顶部状态区比状态标签更重要，需要解释下一步。

```txt
用户可以选择预约时间
用户可以看到金额明细
用户可以提交订单
订单列表能按状态展示订单
订单详情能展示状态、服务、地址、金额和操作按钮
待支付、服务中、待确认、已完成等状态的按钮不同
空状态、加载状态、错误状态完整
底部按钮不被安全区遮挡
```

师傅端：

```txt
师傅可以进入工作台
师傅可以看到待接单、进行中、已完成任务
师傅可以进入任务详情
任务详情能突出预约时间、地址、电话和导航入口
不同任务状态展示不同操作按钮
上传照片组件可用
完成服务前有必要校验和二次确认
```

视觉和体验：

```txt
主流程页面风格一致

长服务名称不撑破卡片

金额展示不换行混乱
底部固定操作栏适配 iPhone 安全区
小屏手机首屏信息不过载
不要出现大面积营销装饰压过服务入口
```

## 15. Day 4 不做清单

```txt
不做完整钱包

不做银行卡绑定

不做会员成长体系

不做服务商完整分佣

不做复杂活动专题
```

这些能力可以参考原型保留入口，但不进入 Day 4 主开发。

## 16. 风险点

- 参考端功能很多，不能被它的完整平台化范围牵着走；Day 4 只服务订单闭环。
- 前端金额展示必须以后端计算为准，不能只在本地算。
- 订单状态必须来自后端状态机，前端只负责展示和触发动作。
- 师傅端操作必须做权限校验，不能只靠隐藏按钮。
- 照片上传后续需要内容安全审核，Day 4 UI 先预留状态。
- 用户端和师傅端在同一小程序内时，入口必须清楚，不能干扰普通用户。


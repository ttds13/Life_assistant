# 生活助手小程序 MVP UI 开发计划

## 1. 文档目的

本文档用于指导生活助手小程序第一版 MVP 的 UI 设计和前端页面开发。

本项目不建议先做完整全量高保真设计稿，也不建议完全边写代码边临时想 UI。更合适的方式是：

```txt
先做核心流程线框图
-> 确定基础视觉规范
-> 做少量关键页面高保真
-> 基于组件库开发页面
-> 每完成一个业务模块就补齐该模块 UI 状态
```

目标是让独立开发者能够快速做出清晰、可信、可商用的生活服务类小程序，而不是把大量时间耗在复杂视觉设计上。

## 2. UI 开发总原则

## 2.1 先流程，后视觉

第一版 UI 的核心不是好看，而是让用户顺利完成：

```txt
找服务
看价格
选时间
填地址
下单
支付
看订单进度
评价或投诉
```

如果流程不清晰，再漂亮的页面也会返工。

## 2.2 优先使用组件库

小程序端建议使用：

```txt
uni-app
Vue3
TypeScript
Wot UI 或 uView Plus
Pinia
Sass
```

管理后台建议使用：

```txt
Vue3
TypeScript
Element Plus
Vben Admin 或同类后台模板
```

第一版不要自己设计太多基础组件。按钮、表单、弹窗、标签、列表、日期选择、上传、步骤条等优先使用组件库。

## 2.3 统一样式，不做页面级随意发挥

第一版只建立一套简单、稳定的视觉规范：

- 统一主色。
- 统一页面背景。
- 统一卡片样式。
- 统一按钮样式。
- 统一表单样式。
- 统一订单状态标签。
- 统一列表布局。
- 统一空状态。
- 统一加载和错误状态。

## 2.4 页面数量要控制

第一版只对关键页面做高保真样张，其他页面按规范复用。

必须重点设计：

```txt
用户端首页
服务详情页
预约下单页
订单详情页
师傅工作台
师傅任务详情页
后台订单列表
后台订单详情和派单弹窗
```

## 3. UI 工作阶段总览

```txt
阶段 UI-0：UI 参考和设计方向确认
阶段 UI-1：信息架构和页面地图
阶段 UI-2：核心流程线框图
阶段 UI-3：基础视觉规范
阶段 UI-4：小程序关键页面高保真
阶段 UI-5：管理后台关键页面规范
阶段 UI-6：组件清单和状态规范
阶段 UI-7：按业务模块开发 UI
阶段 UI-8：真机检查和 UI 走查
阶段 UI-9：后续复杂 UI 能力预留
```

## 4. 阶段 UI-0：UI 参考和设计方向确认

## 4.1 阶段目标

确认产品视觉方向，避免开发过程中反复改风格。

## 4.2 需要做的事

- 找 3 到 5 个生活服务类小程序作为参考。
- 找 2 到 3 个本地服务、家政、维修、上门服务类 App 或小程序参考。
- 截图保存以下页面：
  - 首页
  - 分类页
  - 服务详情页
  - 下单页
  - 订单详情页
  - 我的页面
  - 师傅工作台或类似页面
- 总结参考产品的共同结构。
- 确认本项目 UI 风格关键词。

## 4.3 推荐风格关键词

```txt
清晰
可信
简洁
高效
生活服务
轻运营
弱装饰
强流程
```

## 4.4 不建议的风格

```txt
过度营销化
大面积渐变
大量插画
过多卡片嵌套
复杂动效
花哨图标
暗黑科技风
```

生活服务类小程序应该让用户觉得可靠、价格清楚、服务流程明确。

## 4.5 产出物

```txt
docs/design/references/        # UI 参考截图，可后续创建
docs/design/ui-direction.md    # UI 方向说明，可后续创建
```

## 4.6 验收标准

- 已确定整体 UI 风格。
- 已确定小程序主色倾向。
- 已确定页面结构不走营销落地页风格。
- 已确认第一版主要靠组件库完成。

- 师傅端入口不影响普通用户使用。

## 6. 阶段 UI-2：核心流程线框图

## 6.1 阶段目标

用低保真线框图确认业务流程，不纠结颜色、图标和视觉细节。

## 6.2 必画流程

## 6.2.1 用户下单流程

```txt
首页
-> 服务分类
-> 服务详情
-> 预约下单
-> 确认金额
-> 微信支付
-> 支付结果
-> 订单详情
```

线框图需要明确：

- 服务入口在哪里。
- 价格在哪里展示。
- 用户在哪里选择时间。
- 用户在哪里选择地址。
- 优惠券入口在哪里。
- 支付按钮在哪里。
- 下单失败如何提示。

## 6.2.2 后台派单流程

```txt
后台订单列表
-> 订单详情
-> 选择师傅
-> 确认派单
-> 派单成功
```

线框图需要明确：

- 待派单订单如何突出。
- 管理员如何筛选订单。
- 派单弹窗展示哪些员工信息。
- 派单后状态如何变化。

## 6.2.3 师傅履约流程

```txt
师傅工作台
-> 待接单任务
-> 任务详情
-> 接单
-> 上门打卡
-> 上传照片
-> 完成服务
```

线框图需要明确：

- 今日任务如何展示。
- 接单和拒单按钮在哪里。
- 用户地址和导航入口在哪里。
- 上传照片入口在哪里。
- 完成服务前需要哪些校验。

## 6.2.4 售后流程

```txt
订单详情
-> 发起售后
-> 填写问题
-> 上传凭证
-> 提交工单
-> 查看处理进度
```

## 6.3 线框图工具

可选：

```txt
Figma
即时设计
墨刀
Pixso
纸笔草图
```

如果不擅长 UI，纸笔或低保真工具足够。线框图只解决结构问题。

## 6.4 验收标准

- 四个核心流程都有线框图。
- 每个关键页面有主要区域布局。
- 每个关键操作有明确按钮。
- 每个流程知道成功、失败和返回路径。

## 7. 阶段 UI-3：基础视觉规范

## 7.1 阶段目标

建立一套足够第一版使用的视觉规范，让后续页面可以直接复用。

## 7.2 色彩规范

建议采用偏可信、清爽的生活服务配色。

```txt
主色：#1677FF
主色浅色背景：#EAF3FF
成功色：#16A34A
警告色：#F59E0B
错误色：#EF4444
正文色：#1F2937
次要文字：#6B7280
弱文字：#9CA3AF
边框色：#E5E7EB
页面背景：#F5F7FA
卡片背景：#FFFFFF
```

注意：

- 不要让整个应用都变成单一蓝色。
- 营销活动可以使用少量橙色或红色，但不能喧宾夺主。
- 订单状态颜色要固定，不要每个页面随意配置。

## 7.3 字体层级

小程序端建议：

```txt
页面标题：20px / 600
模块标题：17px / 600
正文主要：15px / 400
正文辅助：13px / 400
说明文字：12px / 400
价格数字：22px / 600
```

后台 Web 建议：

```txt
页面标题：20px / 600
卡片标题：16px / 600
表格正文：14px
说明文字：12px
```

## 7.4 间距规范

小程序端：

```txt
页面左右边距：16px
卡片内边距：16px
模块间距：12px 或 16px
列表项间距：8px 或 12px
按钮高度：44px
底部安全区：适配 safe-area
卡片圆角：8px
```

后台 Web：

```txt
页面内边距：16px 或 24px
表格工具栏间距：12px
搜索表单项间距：12px
弹窗宽度：600px 到 900px
```

## 7.5 状态颜色

订单状态建议固定：

```txt
待支付：warning
待派单：warning
已派单：primary
已接单：primary
上门中：primary
服务中：primary
待确认：warning
已完成：success
已取消：info
退款中：warning
已退款：info
售后中：error
```

## 7.6 验收标准

- 主色、辅助色、状态色已确定。
- 字体层级已确定。
- 间距和卡片规则已确定。
- 订单状态标签颜色已确定。
- 前端可将这些规则写入 Sass 变量或 CSS 变量。

## 8. 阶段 UI-4：小程序关键页面高保真

## 8.1 阶段目标

只对第一版最重要的页面做高保真样张，作为后续开发的视觉基准。

## 8.2 必做高保真页面

## 8.2.1 用户端首页

页面内容：

- 顶部定位城市。
- 搜索框。
- 服务分类宫格。
- 热门服务。
- 优惠活动入口。
- 推荐服务。
- 底部 Tab。

设计重点：

- 服务入口要明显。
- 不做复杂大图英雄区。
- 优惠活动可以占一小块，但不能压过服务入口。
- 首页首屏要让用户知道可以预约哪些服务。

## 8.2.2 服务详情页

页面内容：

- 服务主图。
- 服务名称。
- 起步价或估价说明。
- 服务介绍。
- 服务范围。
- 服务流程。
- 用户评价摘要。
- 底部预约按钮。

设计重点：

- 价格展示必须清楚。
- 预约按钮固定在底部。
- 服务须知不要隐藏太深。
- 评价和服务说明结构清晰。

## 8.2.3 预约下单页

页面内容：

- 服务信息摘要。
- 估价参数。
- 预约时间。
- 服务地址。
- 备注。
- 优惠券入口。
- 会员次数卡入口。
- 金额明细。
- 提交订单按钮。

设计重点：

- 表单分组清晰。
- 金额明细清楚。
- 用户知道每一项是否必填。
- 底部按钮固定。

## 8.2.4 订单详情页

页面内容：

- 当前状态。
- 状态进度。
- 服务信息。
- 预约时间。
- 服务地址。
- 师傅信息。
- 支付信息。
- 服务照片。
- 操作按钮。

设计重点：

- 当前状态必须突出。
- 不同状态显示不同按钮。
- 用户能理解下一步要做什么。

## 8.2.5 师傅工作台

页面内容：

- 今日任务数量。
- 待接单数量。
- 进行中数量。
- 今日收入预估。
- 任务快捷入口。
- 消息提醒。

设计重点：

- 任务优先级清楚。
- 接单入口明显。
- 不做复杂统计图。

## 8.2.6 师傅任务详情页

页面内容：

- 订单状态。
- 服务内容。
- 预约时间。
- 用户地址。
- 导航按钮。
- 用户备注。
- 接单/拒单。
- 打卡。
- 上传照片。
- 完成服务。

设计重点：

- 操作按钮随状态变化。
- 地址和导航非常明显。
- 照片上传流程简单。

## 8.3 可后补高保真页面

```txt
订单列表
我的页面
地址管理
优惠券列表
评价页
售后页
会员次数卡页
```

这些页面可以先按组件库规范开发，不必一开始画高保真。

## 8.4 验收标准

- 6 个小程序关键页面有高保真样张。
- 高保真页面遵守基础视觉规范。
- 关键按钮位置清晰。
- 页面首屏信息不过载。
- 可直接作为前端开发依据。

## 9. 阶段 UI-5：管理后台关键页面规范

## 9.1 阶段目标

后台以效率为主，优先使用成熟管理模板和表格表单组件。

## 9.2 后台设计原则

- 不做营销式首页。
- 不做复杂视觉装饰。
- 表格、筛选、弹窗、详情页要清楚。
- 操作路径要短。
- 关键操作要有二次确认。
- 危险操作要记录原因。

## 9.3 必做页面规范

- 近 7 日收入趋势。

## 9.3.2 订单列表

- 状态筛选。
- 日期筛选。
- 服务类型筛选。
- 员工筛选。
- 表格。
- 批量导出。
- 查看详情。
- 派单操作。

## 9.3.3 订单详情

内容：

- 订单基础信息。
- 用户信息。
- 服务信息。
- 地址信息。
- 支付信息。
- 师傅信息。
- 状态日志。
- 服务照片。
- 售后记录。
- 管理员备注。

## 9.3.4 派单弹窗

内容：

- 当前订单信息摘要。
- 可选员工列表。
- 员工状态。
- 员工今日任务数。

## 9.3.5 服务管理

内容：

- 分类管理。
- 服务项目列表。
- 服务编辑表单。
- 价格配置。
- 上下架状态。
- 服务图片上传。

## 9.4 验收标准

- 后台订单处理路径清楚。
- 管理员 3 步内可以完成派单。
- 详情页信息完整但不混乱。
- 表格筛选项足够第一版运营使用。

## 11. 阶段 UI-7：按业务模块开发 UI

## 11.1 阶段目标

进入编码后，UI 跟随业务模块逐步开发，每个模块完成后都能单独验收。

## 11.2 模块 UI 开发顺序

## 11.2.1 账号与我的模块

页面：

```txt
用户登录
手机号授权
我的页面
员工登录
师傅个人资料
后台登录
```

验收：

- 登录入口清晰。
- 未登录状态友好。
- 普通用户和师傅入口不混乱。

## 11.2.2 服务浏览模块

页面：

```txt
首页
服务分类
搜索
服务详情
```

验收：

- 用户能快速找到服务。
- 服务价格和预约入口明显。
- 搜索结果可用。

## 11.2.3 预约下单模块

页面：

```txt
预约下单
地址列表
地址编辑
优惠券选择入口
支付结果
```

## 12. 阶段 UI-8：真机检查和 UI 走查

## 12.1 阶段目标

保证小程序在真实手机上能正常使用，不出现文字溢出、按钮遮挡、状态混乱等问题。

## 12.2 小程序检查清单

- iPhone 小屏幕显示正常。
- Android 常见屏幕显示正常。
- 底部按钮不被安全区遮挡。
- 长服务名不溢出。
- 长地址不撑破卡片。
- 金额展示不换行混乱。
- 图片上传区域不变形。
- 弱网加载有提示。
- 空数据页面不突兀。
- 错误提示能看懂。
- 弹窗按钮不拥挤。
- 订单状态标签颜色一致。

## 12.3 后台检查清单

- 1366px 宽度下表格可用。
- 表格列不严重挤压。
- 搜索条件不换行混乱。
- 弹窗高度不超屏。
- 详情页分区清楚。
- 危险操作有二次确认。
- 表单校验清楚。
- 导出按钮位置合理。

## 12.4 验收标准

- 真机主流程能完整跑通。
- 关键页面没有明显错位。
- 关键按钮没有被遮挡。
- 长文本不会破坏布局。
- 空状态、错误状态、加载状态都可用。

## 17. 每个页面开发前检查

```txt
这个页面的主要目标是什么？

有哪些状态？
空数据怎么显示？
加载中怎么显示？
接口失败怎么显示？
长文本会不会溢出？
底部按钮会不会被挡住？
```

如果这些问题回答不出来，先不要写页面代码。
